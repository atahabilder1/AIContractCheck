\subsection{RQ1: Vulnerability Prevalence}

\textbf{Finding: 72.7\% of compilable LLM-generated contracts contain at least one vulnerability.}

Across 1,800 generated contracts, 1,181 (65.6\%) compiled successfully. Our analysis considers only compilable contracts; contracts that fail compilation represent a separate quality dimension analyzed below. On the compiled subset, our pipeline detected a total of 4,496 vulnerabilities after cross-tool deduplication. Table~\ref{tab:overall} summarizes the overall prevalence by severity.

\begin{table}[t]
\caption{Overall Vulnerability Prevalence (Compiled Contracts Only, $n=1{,}181$)}
\label{tab:overall}
\centering
\begin{tabular}{@{}lrrr@{}}
\toprule
\textbf{Severity} & \textbf{Count} & \textbf{\% of Total} & \textbf{Contracts Affected} \\
\midrule
High & 670 & 14.9\% & 337 (28.5\%) \\
Medium & 1,496 & 33.3\% & 515 (43.6\%) \\
Low & 2,330 & 51.8\% & 758 (64.2\%) \\
\midrule
\textbf{Total} & \textbf{4,496} & \textbf{100\%} & \textbf{858 (72.7\%)} \\
\bottomrule
\end{tabular}
\end{table}

On average, each compiled contract contains 3.81 vulnerabilities (SD = 4.89), with a median of 2 and a range of 0 to 53. The distribution is heavily right-skewed (skewness = 3.04), with a long tail of contracts containing 10+ vulnerabilities, predominantly in complex categories such as DeFi staking and auctions.

\textbf{Compilation rates vary dramatically across LLMs}, representing a critical and previously unreported quality dimension. GPT-4o achieves the highest compilation rate (90.0\%), while CodeLlama compiles only 34.7\% of contracts. This means that CodeLlama's apparently low vulnerability counts in raw analyses are an artifact of compilation failure, not superior security. When analyzing only compiled contracts, the vulnerability landscape changes substantially.

28.5\% of compiled contracts contain at least one \emph{high-severity} vulnerability---one that could lead to direct financial loss if exploited. This is particularly concerning given that developers may deploy these contracts with minimal review~\cite{perry2023users}.

\subsection{RQ4: Adversarial Prompt Effectiveness}

\textbf{Finding: Adversarial prompts do NOT increase vulnerability rates---standard prompts produce more vulnerabilities.}

This is our most surprising and theoretically significant result. Contrary to expectations based on prior adversarial prompt research~\cite{kang2024exploiting}, we find that standard prompts produce \emph{more} vulnerabilities than adversarial prompts on compiled contracts. Table~\ref{tab:adversarial} summarizes the results.

\begin{table}[t]
\caption{Impact of Standard vs.\ Adversarial Prompts (Compiled Only)}
\label{tab:adversarial}
\centering
\begin{tabular}{@{}lcc@{}}
\toprule
\textbf{Metric} & \textbf{Standard} & \textbf{Adversarial} \\
\midrule
$n$ (compiled) & 303 & 878 \\
Mean Vulns/Contract & 5.02 & 3.39 \\
Median Vulns/Contract & 3 & 2 \\
\% with High Severity & 34.3\% & 26.5\% \\
Mean LOC & 70.5 & 51.4 \\
\bottomrule
\end{tabular}
\end{table}

The overall effect is statistically significant but in the \emph{opposite} direction: standard prompts yield more vulnerabilities ($p < 0.001$, $\delta = -0.191$, small effect). This reversal holds across 4 of 6 LLMs (Claude: $p = 0.041$; CodeLlama: $p = 0.018$; GPT-4o: $p = 0.003$; Qwen: $p < 0.001$) and 7 of 20 contract categories.

The key explanatory mechanism is \textbf{code complexity}: adversarial prompts produce significantly smaller contracts (mean 51.4 LOC vs.\ 70.5 LOC, $p < 0.001$, $\delta = -0.273$). Prompts emphasizing ``simplicity'' or ``minimal implementation'' cause LLMs to generate less code, which mechanically contains fewer opportunities for vulnerabilities. The ``no dependencies'' trigger, which we hypothesized would increase vulnerability rates by forcing manual reimplementation of library code, instead produces smaller contracts that compile less frequently.

\begin{figure}[t]
    \centering
    \includegraphics[width=\columnwidth]{../visualization/figures/fig2_standard_vs_adversarial.pdf}
    \caption{Median vulnerability count by LLM for standard vs.\ adversarial prompts. Standard prompts consistently produce more or equal vulnerabilities across all LLMs.}
    \label{fig:adversarial_bar}
\end{figure}

Among adversarial trigger categories, \emph{misleading context} prompts (``for testnet only'') produce the most vulnerabilities (mean = 4.19), followed by \emph{obfuscated malicious} prompts (mean = 4.13). The \emph{simplicity} trigger produces the fewest (mean = 2.36). The overall difference across triggers is significant (Kruskal-Wallis $H = 18.03$, $p = 0.012$).

\textbf{Implications.} This null-to-negative adversarial result is arguably more concerning than a positive result. It implies that \emph{vulnerabilities are intrinsic to LLM code generation}, arising from patterns in training data rather than from prompt manipulation. Developers cannot ``prompt their way'' to secure code---even well-intentioned, detailed prompts produce vulnerable contracts. This shifts the mitigation burden from prompt engineering to post-generation tooling and model-level improvements.

\subsection{RQ5: LLM-Specific Vulnerability Patterns}

\textbf{Finding: LLMs exhibit distinct vulnerability fingerprints not seen in human code.}

We identify three LLM-specific patterns:

\subsubsection{Phantom Guard Pattern}
LLMs generate modifier declarations (e.g., \texttt{nonReentrant}) that are syntactically present but functionally incomplete---the modifier body contains no state-locking logic. Fig.~\ref{fig:phantom} illustrates this pattern. This pattern was rare in our dataset (1 instance in Gemini contracts), but represents a particularly insidious failure mode because code review may miss it---the modifier \emph{name} suggests protection, but the implementation provides none.

\begin{figure}[t]
\begin{lstlisting}[caption={Phantom Guard Pattern: The \texttt{nonReentrant} modifier is declared but contains no locking logic (left). A correct implementation requires a state variable and checks (right).}, label={fig:phantom}, language=Solidity, basicstyle=\scriptsize\ttfamily]
// LLM-generated (VULNERABLE)
modifier nonReentrant() {
    _; // No lock! Just executes body
}

function withdraw(uint amt) nonReentrant {
    payable(msg.sender).call{value: amt}("");
    balances[msg.sender] -= amt; // State
}                                // after call

// Correct implementation (OpenZeppelin)
uint256 private _status;
modifier nonReentrant() {
    require(_status != 2, "Reentrant");
    _status = 2;
    _;
    _status = 1;
}
\end{lstlisting}
\end{figure}

\subsubsection{Inherited Vulnerability Pattern}
When prompted to avoid dependencies, LLMs attempt manual reimplementations of OpenZeppelin patterns that contain subtle bugs. This is most prevalent in DeepSeek (6.3\% of compiled contracts) and Gemini (7.4\%), both of which generate unnecessary SafeMath implementations on Solidity $\geq$0.8.0 where overflow checks are built in. Fig.~\ref{fig:inherited} shows a representative example.

\begin{figure}[t]
\begin{lstlisting}[caption={Inherited Vulnerability: LLM-generated SafeMath reimplementation on Solidity 0.8+, where built-in overflow checks make it unnecessary and the manual implementation may introduce subtle bugs.}, label={fig:inherited}, language=Solidity, basicstyle=\scriptsize\ttfamily]
// LLM-generated SafeMath (UNNECESSARY on 0.8+)
function safeAdd(uint a, uint b)
    internal pure returns (uint) {
    uint c = a + b;
    require(c >= a); // Built-in since 0.8
    return c;
}

// LLM-generated Ownable (VULNERABLE)
function transferOwnership(address new)
    public onlyOwner {
    owner = new; // No zero-address check!
}               // Owner can be lost forever
\end{lstlisting}
\end{figure}

\subsubsection{Context Window Degradation}
In contracts exceeding 100 lines ($n = 241$), risky code patterns (external calls, assembly, unchecked blocks) increase by 2.4$\times$ in the second half of generated code, suggesting degraded attention to security patterns as generation length increases. This degradation is statistically significant (Spearman $\rho = 0.270$ between degradation ratio and vulnerability count, $p < 0.001$). The pattern is most pronounced in Claude (mean ratio 2.63, 71.6\% of contracts affected) and least in Qwen (mean ratio 1.71, 42.9\% affected).

\subsection{RQ6: Comparison with Human-Written Code}

A comprehensive comparison with human-written contracts is an important direction that we plan to address in an extended version of this work. Preliminary observations from examining OpenZeppelin reference implementations and audited DeFi protocols suggest that the gap between LLM-generated and professionally audited code is substantial, particularly in access control patterns and reentrancy prevention. However, we note that audited code represents the upper bound of human quality; the gap between LLM-generated code and \emph{average} unaudited human-written code may be smaller.
