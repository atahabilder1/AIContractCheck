\section{Discussion}
\label{sec:discussion}
%======================================================================

\subsection{Implications for Developers}

Our results demonstrate that LLM-generated smart contracts should \emph{never} be deployed without comprehensive security auditing. The 64.5\% vulnerability rate among compilable contracts, combined with the presence of high-severity flaws in 26.8\% of contracts, makes unaudited deployment reckless. We recommend the following guidelines based on our findings:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Always use multi-tool analysis.} Run generated contracts through multiple static analysis tools (at minimum, Slither and one symbolic execution tool) before deployment. No single tool catches all vulnerability types.
    \item \textbf{Do not rely on prompt engineering for security.} Our adversarial analysis shows that vulnerabilities are intrinsic to LLM code generation, not prompt-dependent. Even carefully crafted prompts produce vulnerable code. Post-generation auditing is the only reliable mitigation.
    \item \textbf{Prefer models with high compilation rates.} GPT-5 (91.0\%), GPT-4o (90.0\%), and Qwen (79.0\%) produce the most consistently compilable code. CodeLlama's low compilation rate (34.7\%) makes it unsuitable for production smart contract generation. GPT-5 also achieves the lowest vulnerability density (2.22 vulns/100 LOC), making it the strongest choice for security-sensitive contract generation.
    \item \textbf{Exercise extreme caution with complex categories.} DeFi staking, DEX/AMM, and yield aggregator contracts exhibit 2--3$\times$ higher vulnerability rates than standard tokens. These categories should receive proportionally more scrutiny.
    \item \textbf{Verify security pattern completeness.} The phantom guard pattern (Section~\ref{sec:results}) demonstrates that syntactic presence of security constructs does not guarantee functional protection. Manually verify that modifiers and guards contain correct logic.
    \item \textbf{Review the second half of long contracts more carefully.} Context window degradation causes 4.1$\times$ higher risky pattern density in the latter portions of complex contracts (>100 LOC).
\end{enumerate}

\subsection{Compilation Quality Analysis}

Of 2,400 generated contracts, 834 (34.8\%) failed to compile. Root causes are dominated by semantic/type errors (68.5\%), followed by import/dependency issues (13.3\%), output format contamination such as markdown code fences (7.3\%), syntax errors (6.8\%), and compiler version incompatibilities (4.1\%). LLM-specific patterns are revealing: CodeLlama generates deprecated pre-0.8.x syntax, Gemini suffers from markdown contamination, Claude references bleeding-edge compiler versions, and GPT-4o's failures are almost purely semantic (80\%), indicating the highest Solidity fluency.

\subsection{Implications for LLM Providers and the Security Community}

Our findings suggest actionable improvements: (1)~\textbf{Security-focused fine-tuning}---incorporate verified, audited smart contract code (e.g., OpenZeppelin) as high-priority training data and downweight unaudited Etherscan code~\cite{bai2022training}; (2)~\textbf{Post-generation validation}---integrate automatic security scanning that flags common patterns (reentrancy, missing access control) before presenting code to users; (3)~\textbf{Security warnings}---include explicit disclaimers when generating financial contracts; and (4)~\textbf{Context window management}---address the degradation pattern by maintaining security-relevant context throughout long generations.

For the security community, our work motivates \textbf{LLM-aware security tools} that target the phantom guard, inherited vulnerability, and context degradation patterns we identified. Our dataset of 2,400 labeled contracts can serve as a benchmark. The finding that vulnerabilities are intrinsic (not prompt-dependent) motivates research into training data curation for security-critical code generation.

\subsection{Limitations}

We acknowledge several limitations: (1)~\textbf{Model evolution}---LLM behavior evolves with updates; our results reflect specific versions, though structural patterns (phantom guards, context degradation) likely persist. (2)~\textbf{Prompt representativeness}---our 300 prompts across 20 categories may not fully capture real-world prompting patterns. (3)~\textbf{Tool limitations}---static analysis produces false positives and negatives; we mitigate through multi-tool analysis and deduplication, with manual validation of 120 contracts underway. (4)~\textbf{Model coverage}---eight LLMs are studied; extending to additional models is future work. (5)~\textbf{Language scope}---we focus on Solidity; generalization to Rust/Solana, Move/Aptos requires further investigation. (6)~\textbf{Human baseline bias}---our 241 Etherscan contracts represent production-grade code but may not capture the full distribution of human-written Solidity; the 54.8\% compilation rate introduces selection bias toward simpler contracts. (7)~\textbf{Compilation bias}---our compiled-only analysis (65.2\%) may miss vulnerability patterns in non-compilable contracts.

\subsection{Threats to Validity}

\textbf{Internal}: Prompt design may bias results; we mitigate this with diverse prompts across complexity levels/categories and a consistent system prompt across all models. \textbf{External}: Results may not generalize to multi-turn interactive sessions; we focus on single-shot generation as the most common and highest-risk use case. \textbf{Construct}: Vulnerability counts may not reflect exploitability; we report severity-stratified results and LOC-normalized rates to control for contract size.

\section*{Data Availability \& Artifact}
%======================================================================

Our complete reproducibility package is publicly available at: \url{https://github.com/[anonymized-for-review]}. The artifact includes: (1)~all 2,400 LLM-generated contracts with prompt metadata, (2)~241 human baseline contracts (Etherscan-verified, forge-flattened), (3)~raw and aggregated analysis results from Slither, Mythril, and Semgrep in structured JSON/CSV, (4)~all analysis scripts (dataset generation, vulnerability pipeline, statistical analysis, figure generation), (5)~the CWE-mapped vulnerability taxonomy, and (6)~instructions for end-to-end reproduction. Model versions, API configurations, and compiler versions are documented to enable exact replication.

%======================================================================
\section*{Ethical Considerations}
%======================================================================

Our adversarial prompts are designed to study LLM behavior, not to enable attacks. We follow responsible disclosure practices: findings regarding specific LLM vulnerabilities were reported to respective providers prior to publication. Our dataset contains only synthetically generated contracts; no deployed contracts were tested or exploited. The adversarial prompt categories we document are constructed from patterns already discussed in the security literature~\cite{kang2024exploiting, perez2022ignore}, and we do not introduce novel attack techniques.
